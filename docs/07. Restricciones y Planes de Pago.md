# Current Calendar

En este documento se detalla la creación y modificación de la implementación de planes de pago en la aplicación Current Calendar

## Metodología empleada

Debido a la arquitectura del proyecto, para la creación y gestión de las limitaciones de los planes de pago se empleará un enfoque nativo a Django Rest Framework, con el objetivo de tener todas las restricciones centralizadas de forma que sean facilmente modificables en caso de que los planes de pago muten.

## entitlements.py

Se trata de la única fuente de verdad para las restricciones de los planes de pago. Contiene:
- Una clase Plans en la que se definirán los planes de pago que tenemos en la aplicación.
- Un diccionario PLAN_FEATURES en el que escribiremos las restricciones de cada plan: 

- Un método auxiliar para obtener el plan del usuario loggeado: 
```python
def get_user_features(user):
``` 

## Aplicar las restricciones

Tras definir nuestras restricciones, podemos separarlas en dos tipos diferenciados: 

### Restricciones que bloquean features

Para aplicar este tipo de restricciones a nuestras features, debemos crear un decorador que aplicaremos a la función correspondiente en el archivo  ```permissions.py```, en el que llamaremos a la función que definimos en ```entitlements.py``` para extraer la información correspondiente. 

#### Ejemplo - Restricción de creación de calendarios:
Crearemos el decorador de esta forma:

```python
class CanCreateCalendar(permissions.BasePermission):
    message = "You have reached the maximum amount of calendars allowed for your plan."

    def has_permission(self, request, view):
        if not request.user.is_authenticated:
            return False

        requested_privacy = request.data.get('privacy', 'PRIVATE')
        user_features = get_user_features(request.user)

        if requested_privacy == 'PUBLIC':
            calendar_limit = user_features['max_public_calendars']
            self.message = "You have reached the maximum amount of public calendars allowed for your plan."

        elif requested_privacy == 'PRIVATE': 
            calendar_limit = user_features['max_private_calendars']
            self.message = "You have reached the maximum amount of private calendars allowed for your plan."
        
        else:
            return True

        if calendar_limit == math.inf:
            return True
        
        calendars_count = Calendar.objects.filter(creator=request.user, privacy=requested_privacy).count()

        return calendars_count < calendar_limit
 ```

 Lo aplicaremos así a la función: 
 ```python

@permission_classes([IsAuthenticated, CanCreateCalendar])

 ```
### Restricciones que cambian el comportamiento de una feature

Otras features, como el radar cambian su funcionamiento según el plan del usuario (enseñan solo los eventos del día de hoy en el caso del plan gratuito y los de las próximas 48 horas en los planes de pago). Un decorador no sería de ayuda en este tipo de casos, puesto que bloquearía el endpoint y devolvería un error ```403 FORBIDDEN```. 

La solución en estos casos, es emplear de forma agnóstica los datos que obtenemos de ```get_user_plan``` en la feature.
